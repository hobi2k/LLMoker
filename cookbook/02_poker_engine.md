# 02. 포커 엔진

[목차로 돌아가기](README.md)

이 장은 `llmoker/backend/poker_engine.py`를 중심으로, **LLMoker의 실제 포커 규칙과 상태 변화**를 설명한다.

이 파일은 게임의 진실이다.  
대사와 UI가 아무리 어색해도 엔진이 맞으면 판은 맞게 굴러간다.  
반대로 엔진이 틀리면 화면이 예뻐도 게임은 틀린다.

## 1. 핵심 데이터 구조

### 1.1 `PlayerState`

역할:

- 플레이어 하나의 상태를 담는다.

주요 필드:

- `name`
- `stack`
- `hand`
- `folded`

입력:

- 플레이어 이름
- 시작 칩 수

출력:

- 별도 반환값은 없고, 매치 내부에서 상태 보관 구조로만 쓰인다.

의미:

플레이어와 봇을 같은 구조로 다루기 때문에, 베팅과 드로우 로직이 같은 코드 경로를 공유할 수 있다.

### 1.2 `PokerMatch`

역할:

- 2인 5드로우 포커 한 매치 전체를 관리한다.

핵심 필드:

- `player`, `bot`
- `deck`
- `pot`
- `phase`
- `current_actor`
- `current_bet`
- `player_contribution`, `bot_contribution`
- `raises_in_round`
- `consecutive_checks`
- `round_over`
- `action_log`
- `public_log`
- `round_summary`
- `latest_feedback`
- `llm_agent`
- `memory_manager`

여기서 가장 중요한 구분:

### `public_log`

공개 사건만 들어간다.

예:

- 상대가 체크했다
- 플레이어는 10칩 베팅했다
- 사야는 10칩 콜했다
- 사야는 총 20칩까지 레이즈했다
- 드로우 단계로 넘어갔다
- 쇼다운 결과가 나왔다

이 로그는:

- 대사 프롬프트
- 공개 상태 조회
- UI 공개 로그 표시

의 재료가 된다.

### `action_log`

내부 디버그와 상세 추적용이다.

예:

- LLM 행동 선택 로그
- 카드 교체 내부 판단
- 디버그용 이유 문장

이 로그는 QA와 디버깅용이다.  
여기 내용을 `public_log`로 흘리면 대사가 바로 오염된다.

## 2. 라운드 시작

### `start_new_round()`

기능:

- 덱 생성
- 손패 분배
- 앤티 차감
- 팟 초기화
- 로그 초기화
- 첫 베팅 페이즈 진입

입력:

- 없음

출력:

- 라운드 시작 직후 UI와 상태창에 보여줄 메시지 목록

바꾸는 상태:

- `deck`
- `player.hand`, `bot.hand`
- `player.stack`, `bot.stack`
- `pot`
- `phase`
- `action_log`, `public_log`
- 각종 베팅 카운터

감사 포인트:

- 라운드 시작마다 이전 라운드 카드가 남아 있지 않은가
- 앤티와 팟이 정상적으로 초기화되는가
- `public_log` 첫 줄이 공개 사건만 담는가

## 3. 합법 행동 계산

### `_get_available_actions(actor_name)`

기능:

- 현재 액터 기준으로 합법 행동 목록을 계산한다.

입력:

- `actor_name`: `"player"` 또는 `"bot"`

출력:

- 합법 행동 문자열 목록

읽는 상태:

- 현재 베팅액
- 현재 액터 기여액
- 칩 잔액
- 레이즈 횟수

바꾸는 상태:

- 없음

자주 나는 문제:

- `fold`가 빠진다
- `fold`가 두 번 들어간다
- `check`가 나와야 할 때 안 나온다
- `bet`와 `raise`가 잘못된 타이밍에 나온다

이 함수가 틀리면 UI 버튼도 틀리고, LLM이 틀린 행동을 뽑은 것처럼 보일 수도 있다.

### `get_player_available_actions()`

기능:

- 현재 플레이어 차례일 때만 플레이어 행동 목록을 계산한다.

입력:

- 없음

출력:

- 플레이어 행동 문자열 목록

읽는 상태:

- 현재 페이즈
- 현재 액터

바꾸는 상태:

- 없음

의미:

UI는 이 함수 결과를 그대로 믿고 버튼을 만든다.  
그래서 플레이어 버튼 문제는 먼저 이 함수와 `_get_available_actions()`를 같이 봐야 한다.

## 4. 플레이어 행동 적용

### `resolve_player_action(action)`

기능:

- 플레이어 행동을 실제 상태에 반영한다.
- 필요하면 봇 턴까지 연달아 진행한다.

입력:

- `action`: `fold`, `check`, `bet`, `call`, `raise` 중 하나

출력:

- 이번 입력으로 발생한 메시지 목록

읽는 상태:

- 현재 페이즈
- 현재 액터
- 합법 행동 목록

바꾸는 상태:

- 팟
- 칩
- 현재 베팅
- 현재 액터
- 로그
- 경우에 따라 페이즈

중요:

이 함수는 단순 버튼 처리기가 아니다.  
이 함수가 **포커 상태를 커밋**한다.

### `_apply_betting_action(actor_name, action)`

기능:

- 체크, 베팅, 콜, 레이즈, 폴드를 실제로 처리한다.

입력:

- `actor_name`
- `action`

출력:

- 행동 처리 로그 목록

읽는 상태:

- 현재 베팅액
- 콜 금액
- 칩 잔액

바꾸는 상태:

- `pot`
- `player.stack`, `bot.stack`
- `current_bet`
- `consecutive_checks`
- `raises_in_round`
- `public_log`, `action_log`

감사 포인트:

- 체크 두 번이면 베팅 라운드가 정확히 닫히는가
- 베팅, 콜, 레이즈 시 칩과 팟이 정확히 이동하는가
- 폴드 시 즉시 종료 정산이 되는가

### `_run_bot_turns()`

기능:

- 현재 액터가 봇일 동안 봇의 판단과 행동 적용을 이어서 처리한다.

입력:

- 없음

출력:

- 봇 행동과 페이즈 전환으로 생긴 메시지 목록

읽는 상태:

- 현재 페이즈
- 현재 액터
- 봇 모드
- 현재 손패와 공개 상태

바꾸는 상태:

- 팟
- 칩
- 로그
- 현재 액터
- 페이즈

자주 보이는 착시:

플레이어 행동 로그가 약하거나 안 찍히면,  
이 함수 때문에 봇이 혼자 두 번 연속 행동한 것처럼 보일 수 있다.

실제로는 아래 둘을 구분해야 한다.

- 엔진이 진짜 두 번 잘못 돌았는가
- 플레이어 행동이 로그에 약해서 그렇게 보였는가

## 5. 드로우

### `resolve_draw_phase(selected_discards)`

기능:

- 플레이어 카드 교체 적용
- 봇 카드 교체 판단과 적용
- 다음 카드 뽑기
- 다음 베팅 페이즈 진입

입력:

- 플레이어가 버릴 카드 인덱스 목록

출력:

- 드로우 처리 결과 메시지 목록

읽는 상태:

- 현재 손패
- 덱
- 최대 교체 장수
- 봇 모드

바꾸는 상태:

- 플레이어 손패
- 봇 손패
- 덱
- 로그
- 페이즈

가장 위험한 감사 포인트:

- 봇의 내부 LLM 카드 교체 로그가 `public_log`로 새는지

이게 새면:

- 대사 프롬프트 오염
- 행동 판단 오염
- 회고 오염

세 가지가 동시에 일어난다.

## 6. 종료와 요약

### `_advance_after_betting()`

기능:

- 베팅 라운드 종료 후 다음 페이즈로 이동한다.

입력:

- 없음

출력:

- 페이즈 전환 메시지 목록

의미:

- `betting1` 종료 -> `draw`
- `betting2` 종료 -> 쇼다운

### `_finish_by_fold(winner)`

기능:

- 폴드 종료 시 승자 정산과 종료 상태 반영

입력:

- 승자 `PlayerState`

출력:

- 반환값 없음

바꾸는 상태:

- 승자 스택
- `phase`
- `round_over`
- `round_summary`

### 쇼다운 관련 조회 함수

- `get_round_result_title()`
- `get_round_result_message()`
- `get_match_result_message()`

이 함수들은 엔진 내부 상태를 화면용 문장으로 바꿔준다.  
결과 화면이 이상하면 레이아웃뿐 아니라 이 조회 함수가 어떤 문장을 주는지도 봐야 한다.

## 7. 스냅샷

### `to_snapshot()`

기능:

- 현재 매치 상태를 저장 가능한 사전으로 직렬화

입력:

- 없음

출력:

- 세이브용 스냅샷 사전

중요 필드:

- 손패
- 칩
- 팟
- 현재 페이즈
- 로그
- LLM 관련 설정
- `memory_snapshot`

### `from_snapshot(config, memory_manager, replay_logger, snapshot)`

기능:

- 스냅샷에서 `PokerMatch` 재구성

입력:

- 설정 객체
- 기억 관리자
- 리플레이 저장소
- 스냅샷 사전

출력:

- 복원된 `PokerMatch`

감사 포인트:

- 세이브 복원 시 기억까지 같이 복원되는가
- 복원 뒤 LLM 설정이 현재 설정과 어긋나지 않는가

다음 장: [03. LLM 백엔드와 Qwen-Agent](03_llm_backend.md)
- `current_actor`를 항상 `player`로 두고 첫 번째 베팅을 시작한다.

중요한 규칙:

- 매 라운드의 첫 번째 베팅은 항상 플레이어가 먼저 시작한다.
- 드로우에서도 플레이어가 먼저 교체를 고르고, 그 다음에 봇이 교체한다.
- 두 번째 베팅도 다시 플레이어가 먼저 시작한다.

### 베팅 용어 통일

이 프로젝트에서 화면, 로그, 도움말은 아래 표현을 같은 의미로 쓴다.

- `체크`
  - 현재 테이블 베팅이 0칩일 때 그냥 넘기는 행동
- `베팅 10칩`
  - 현재 테이블 베팅이 0칩일 때 처음으로 10칩을 거는 행동
- `콜 n칩`
  - 현재 테이블 베팅을 맞추기 위해 부족한 금액 `n칩`을 내는 행동
- `총 n칩 레이즈`
  - 현재 콜 금액에 고정 베팅 단위 10칩을 더해 총액 `n칩`까지 올리는 행동
- `폴드`
  - 그 라운드를 즉시 포기하는 행동
