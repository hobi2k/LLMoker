# LLMoker

## 한 줄 소개

`LLMoker`는 Ren'Py 기반 2인 5드로우 포커 게임에 로컬 LLM 상대 AI를 결합한 프로젝트다.  
플레이어는 정해진 버튼만 누르는 것이 아니라, 공개 정보만으로 심리를 읽고 대응하는 상대와 포커 한 판을 주고받게 된다.

## 프로젝트 개요

이 프로젝트는 “LLM을 게임에 붙이면 무엇이 실제로 재미있어지는가”를 확인하기 위해 시작했다.  
규칙이 너무 복잡한 카드게임보다, 숨겨진 정보와 심리전이 분명한 5드로우 포커가 LLM 상대 AI와 잘 맞는다고 판단했다.

핵심 목표는 세 가지였다.

- 정확한 포커 규칙과 상태 전이
- 공개 정보만 보고 판단하는 로컬 LLM NPC
- Ren'Py 위에서 자연스럽게 이어지는 메뉴, 연출, 나레이션

## 왜 5드로우 포커인가

5드로우 포커는 구조가 간단하지만 심리전은 뚜렷하다.

- 손패는 비공개다.
- 공개되는 정보는 체크, 베팅, 콜, 레이즈, 폴드, 교체 수처럼 제한적이다.
- 그래서 상대의 행동 패턴을 보고 추론하는 과정이 자연스럽다.
- 행동 종류가 비교적 적어서 LLM 프롬프트를 통제하기 쉽다.

즉, 규칙 기반 엔진과 LLM 판단을 같이 검증하기 좋은 장르다.

## 구현 범위

현재 구현 범위는 아래와 같다.

- 2인 5드로우 포커 규칙 구현
- 앤티 고정, 첫 베팅, 드로우, 두 번째 베팅, 쇼다운 흐름 구현
- 플레이어 선공, 상대 후공 구조
- `스크립트봇`과 `LLM NPC` 전환 지원
- 로컬 `Qwen3-4B-Instruct-2507` 기반 행동 선택, 카드 교체 판단, 정책 회고
- Ren'Py 메뉴, HUD, 결과 화면, 도움말, 시스템 오버레이 구성
- SQLite 기반 메모리, 리플레이, 저장 구조
- 디버그 로그와 QA 문서 기반 검수 체계

## 시스템 구조

프로젝트는 크게 세 층으로 나뉜다.

### 1. 화면과 흐름

`llmoker/game/*.rpy`

- Ren'Py 라벨 흐름
- 메뉴와 HUD
- 게임 진행 화면
- 행동/단계 시스템 나레이션

### 2. 포커 엔진

`llmoker/backend/*.py`

- 카드 덱 생성
- 족보 평가
- 베팅/콜/레이즈/폴드 처리
- 드로우 처리
- 쇼다운 및 라운드 종료
- 메모리/리플레이 저장

### 3. LLM 런타임

`llmoker/backend/llm/*.py`

- 행동 프롬프트 구성
- 카드 교체 프롬프트 구성
- 정책 회고 프롬프트 구성
- Qwen-Agent 기반 함수 호출
- transformers 런타임 IPC 연결

## LLM NPC 설계

LLM NPC는 플레이어 손패를 보지 않는다.  
오직 아래 공개 정보만 보고 판단한다.

- 현재 페이즈
- 팟 크기
- 현재 베팅액
- 콜 금액
- 허용 행동
- 최근 공개 로그
- 자기 손패와 현재 족보
- 저장된 단기/장기 회고

현재 역할 분리는 아래처럼 고정한다.

- `self`: 사야
- `opponent`: 플레이어

이렇게 용어를 강하게 나누는 이유는, 작은 모델에서 주체 혼선이 생기면 행동과 회고가 바로 망가지기 때문이다.

## 정책 회고 방식

라운드가 끝나면 LLM이 이번 판을 회고하고, 다음 판 판단에 들어갈 짧은 전략 문장을 만든다.

현재 회고는 다음 세 필드로 정리한다.

- `short_term`
- `long_term`
- `strategy_focus`

이 회고는 다음 라운드 행동과 드로우 프롬프트에 다시 들어간다.  
다만 회고 품질이 낮으면 행동 전체를 오염시킬 수 있어서, 역할 용어와 사실 검증 규칙을 계속 다듬고 있다.

## 드로우 제약

초기 구현에서는 LLM이 원페어, 투페어, 트리플 같은 완성 족보를 스스로 깨버리는 문제가 있었다.  
이 문제를 막기 위해, 현재는 엔진 수준에서 **완성 족보 구성 카드 보호 인덱스**를 계산해 교체 직전에 차단한다.

예를 들어:

- 원페어면 페어 두 장은 버릴 수 없다.
- 투페어면 네 장 모두 버릴 수 없다.
- 트리플이면 세 장을 버릴 수 없다.
- 스트레이트/플러시 이상이면 손패 전체를 보호한다.

즉 프롬프트 지시만 믿지 않고, 규칙은 엔진이 직접 보장한다.

## 개발 중 핵심 이슈

이 프로젝트에서 특히 어려웠던 지점은 아래였다.

### 1. LLM이 사실을 틀리게 회고하는 문제

- 승패를 뒤집어 말함
- 이번 판에 없는 족보를 말함
- 패배했는데 승리처럼 서술함

이 문제를 줄이기 위해:

- 역할 용어를 `self/opponent`로 고정
- 공개 사실을 구조화해서 전달
- 회고 결과를 후처리와 검증으로 걸러냄

### 2. 드로우가 완성 족보를 깨는 문제

프롬프트만으로는 막히지 않아서 엔진 하드 제약으로 해결했다.

### 3. 체크/폴드/쇼다운 흐름이 사용자가 보기엔 갑작스럽게 느껴지는 문제

엔진은 맞더라도, 화면에서 “왜 지금 라운드가 끝났는지”를 바로 이해할 수 있어야 했다.  
그래서 행동 나레이션, 도움말, 결과 화면 문구를 계속 다듬고 있다.

## 현재 상태 요약

현재 버전은 “포커 엔진 + 로컬 LLM NPC + 연출/UI + 저장/회고 구조”가 모두 연결된 상태다.  
다만 LLM 정책 회고와 행동 이유의 품질은 계속 다듬고 있으며, 작은 모델에서 생기는 역할 혼선과 전략 오염을 줄이는 작업이 진행 중이다.

정리하면 지금 이 프로젝트는:

- 단순한 데모가 아니라 실제 플레이 가능한 게임 구조를 갖췄고
- LLM을 게임 규칙 안에서 통제하려는 실험이 포함되어 있으며
- 화면, 자산, 엔진, LLM, QA 문서까지 전체 제작 흐름이 연결된 상태다.
