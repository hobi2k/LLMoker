# 대사 시스템 설계

## 1. 목적

`LLMoker`는 포커 엔진만 있는 게임이 아니라, 캐릭터성과 심리전 연출이 같이 돌아가는 구조를 목표로 한다.

그래서 각 라운드의 시작과 종료, 그리고 베팅/드로우 같은 핵심 지점마다 캐릭터 대사가 들어가야 한다.

## 2. 현재 구현 위치

현재 스크립트 대사 레이어는 아래 파일에 들어 있다.

- [poker_dialogue.rpy](llmoker/game/poker_dialogue.rpy)
- [poker_minigame.rpy](llmoker/game/poker_minigame.rpy)
- [prompts.py](llmoker/backend/llm/prompts.py)
- [agent.py](llmoker/backend/llm/agent.py)

## 3. 현재 이벤트 훅

현재 대사는 아래 이벤트에서 호출된다.

1. `match_intro`
   - 매치 시작 전 첫 진입 대사
2. `round_start`
   - 새 라운드 시작 직후
3. `betting`
   - 체크, 베팅, 콜, 레이즈, 폴드 이후 반응
4. `draw`
   - 플레이어가 교체 선택을 끝낸 직후
5. `round_end`
   - 라운드 결과가 확정된 직후
6. `match_end`
   - 한쪽 스택 부족으로 더 이상 라운드를 열 수 없을 때

## 4. 현재 구조

현재는 이벤트 훅은 Ren'Py에 있고, `LLM NPC` 모드에서는 `Qwen-Agent + transformers` 런타임이 대사를 직접 만든다.

- `_dialogue_event_lines(event_name, messages=None)`
  - 이벤트별 대사 후보를 고른다.
- `play_dialogue_event(event_name, messages=None)`
  - 선택된 대사를 Ren'Py 대사창에 순서대로 출력한다.

시각 연출 규칙:

- 대사 이벤트는 별도 정적 캐릭터 이미지를 띄우지 않는다.
- 현재 `main.webm`, `normal.webm`, `lost.webm`, `win.webm`는 `Wan 2.2` 기반으로 생성한 영상 자산이다.
- 포커 기본 배경은 `images/minigames/normal.webm`
- 라운드 종료 배경은 플레이어 승리 시 `images/minigames/lost.webm`, NPC 승리 시 `images/minigames/win.webm`
- 메인 메뉴 배경은 `game/gui/main.webm`
- 현재 GUI 기준 해상도는 `1024x576`이며, 영상과 포커 UI는 같은 비율로 축소해 표시한다.
- 게임 시작 시 검은 화면 없이 바로 `normal.webm` 위에서 대사가 시작된다.
- 라운드 종료 시에는 먼저 `lost.webm` 또는 `win.webm` 위의 결과 화면이 열리고, 그 상태에서 `round_end` 대사가 이어진다.

즉, 지금은 스크립트봇 모드에서는 규칙 기반 대사를 쓰고, LLM NPC 모드에서는 같은 이벤트 위치에 LLM 생성 대사를 직접 연결한 상태다.

대사 심리전의 정보 경계도 행동 판단과 동일하다.

- 대사 생성도 공개된 베팅/체크/레이즈/폴드/교체 정보만 사용해야 한다.
- 플레이어 실제 손패는 대사 생성 프롬프트에 포함하면 안 된다.

### 4.1 현재 실제 연결 상태

여기서 가장 중요한 건 “구조만 있는지”, “실제로 호출되는지”를 분리해서 보는 것이다.

현재 상태는 아래와 같다.

- `play_dialogue_event(...)`
  - 실제 게임 중 호출된다.
- `_dialogue_event_lines(...)`
  - 스크립트봇 모드에서 사용할 규칙 기반 대사를 제공한다.
- `build_dialogue_prompt(...)`
  - 실제 대사 생성용 프롬프트를 만든다.
- `LocalLLMAgent.generate_dialogue(...)`
  - 실제로 외부 transformers 런타임을 호출한다.

즉, 현재 대사 시스템은:

- 이벤트 훅은 실제로 동작함
- LLM 대사 생성 경로가 실제로 연결됨
- 스크립트봇 모드에서는 규칙 기반 대사를 사용함
- LLM NPC 모드에서 대사 생성 실패 시에는 스크립트 대사로 숨기지 않고 시스템 오류 문구를 보여줌
- LLM 대사나 오류 문구는 Ren'Py `say` 전에 `%`, `[]`, `{}`를 모두 이스케이프해 치환/텍스트 태그 해석으로 죽지 않게 한다

## 5. LLM 연결 방향

현재는 아래 구조로 동작한다.

1. 이벤트 발생
2. 현재 포커 상태 수집
3. 캐릭터 기억과 최근 로그 수집
4. `backend/llm/tasks.py`가 대사용 사건 문맥과 감정 힌트를 만든다.
5. `backend/llm/prompts.py`로 프롬프트 구성
6. `backend/llm/agent.py`가 외부 런타임에 대사 생성 요청
7. `backend/llm/runtime.py`가 `dialogue_agent`를 통해 대사를 생성
8. 성공 시 생성 대사 출력
9. LLM NPC 모드 실패 시 시스템 오류 문구 표시

즉, 대사 시스템은 아래 구조다.

`LLM NPC 모드: LLM 대사 우선 -> 실패 시 시스템 오류 표시`

현재 프롬프트 빌더 기준:

- `build_action_prompt(...)`
  - 행동 선택용 프롬프트
- `build_dialogue_prompt(...)`
  - 대사 생성용 프롬프트

### 5.1 대사 프롬프트 입력 요소

`build_dialogue_prompt(...)`는 아래 입력을 받도록 설계되어 있다.

- `event_name`
  - 어떤 대사 이벤트인지
- `recent_feedback`
  - 최근 전략 피드백
- `long_term_memory`
  - 장기 기억
- `recent_log`
  - 최근 공개 게임 로그
- `result_summary`
  - 라운드 결과 요약
- `emotion_hint`
  - round_end/match_end에서 이김/짐/무승부에 따라 넣는 감정 힌트
- `player_name`
  - 실제로 말을 거는 상대 이름
- `bot_name`
  - 현재 대사를 생성하는 NPC 이름

즉 현재 대사 경로는, 많은 상태를 장황하게 주는 대신:

`이벤트 종류 + 최근 공개 사건 한 줄 + 감정 힌트 + 화자/청자 관계`

를 중심으로 심리전 대사를 만들도록 좁혀 둔 상태다.

### 5.2 현재 동작 규칙

- 대사 생성에도 플레이어 손패는 노출하지 않는다.
- 최근 공개 로그는 날것으로 길게 붙이지 않고, 대사용 사건 문장 한 줄로 다시 써서 사용한다.
- 플레이어에게 직접 말하는 2인칭 대사로 제한한다.
- 직전 공개 행동이나 결과 한 가지에 직접 반응하도록 강하게 유도한다.
- 사야의 말투가 드러나도록, 짧고 도발적이지만 여유 있는 대사를 우선한다.
- 설명문보다 실제 한국어 대화처럼 들리는 짧은 반응을 우선한다.
- 번역투보다 자연스러운 한국어 화법이 나오도록 프롬프트를 구성한다.
- 장면 설명 라벨을 길게 나열하지 않고, 방금 공개된 사건 한 줄만 직접 넣는다.
- 대사 프롬프트는 `누가 누구에게 지금 던질 한마디인지`를 먼저 정하고, 사건 설명은 한 줄만 붙여서 프롬프트 문장 자체를 따라 말하지 않게 유지한다.
- `round_end`, `match_end`는 승패 결과를 읽고 기쁨/우쭐함/분함 같은 감정 힌트를 따로 준다.
- 생성 대사는 최대 두 줄로 정리한다.
- 출력 실패 시 LLM NPC 모드에서는 시스템 오류 문구를 그대로 보여준다.
- 대사 생성 성공 시 내부 로그에 `[LLM NPC] ... 대사 생성` 형식으로 남긴다.
- 대사 생성 결과는 터미널에도 `[LLMoker][DEBUG] ...` 형식으로 출력해 검증한다.
- 런타임은 대사 문체를 손보지 않는다. 실제 모델 문장은 그대로 두고, 메타 계획문처럼 보이는 응답만 실패로 처리한다.

## 6. 현재 설계 원칙

- 포커 규칙 엔진과 대사 레이어는 분리한다.
- 대사는 결과를 설명하거나 심리전을 강화해야지, 규칙 판정을 대신하면 안 된다.
- 베팅/드로우/라운드 종료처럼 감정 변화가 큰 지점에만 넣는다.
- UI 흐름을 끊지 않도록 각 이벤트의 대사 수는 짧게 유지한다.
- 대사 이벤트는 현재 상태에 맞는 영상 배경 위에서 재생한다.

## 7. 다음 확장 포인트

- 캐릭터별 페르소나 파일 연동
- 승패와 관계도에 따른 분기 대사
- 이전 라운드 기억 반영
- 블러프 성공/실패에 대한 장기 피드백 반영
- 쇼다운 직전 긴장 연출
- 역할과 말투 같은 고정 규칙은 시스템 메시지에 두고, 이벤트와 최근 공개 사건 같은 가변 정보만 사용자 프롬프트에 둔다.
- 대사 프롬프트는 장면 설명문보다 상대에게 바로 던질 한마디를 우선하게 짧게 유지한다.
- 런타임 후처리는 메타 문장 제거만 담당하고, 어색한 한국어를 억지로 고쳐 쓰지는 않는다.
- 행동, 카드 교체, 정책 회고 프롬프트에는 캐릭터 RP 문구를 넣지 않고 판단 규칙과 출력 형식만 둔다.
