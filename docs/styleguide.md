# LLMoker 스타일 가이드

## 1. 목적

이 문서는 `LLMoker` 코드와 문서를 수정할 때 따라야 하는 작성 기준을 정리한다.
핵심 목표는 두 가지다.

- 처음 보는 사람이 파일을 열었을 때도 흐름을 따라갈 수 있어야 한다.
- 나중에 리팩토링하거나 디버깅할 때, 어디가 규칙이고 어디가 연출인지 바로 구분되어야 한다.

## 2. 기본 원칙

- 코드는 짧게 쓰는 것보다 읽히게 쓰는 것을 우선한다.
- 이름은 줄여 쓰기보다 역할이 드러나게 쓴다.
- 한 함수는 한 가지 책임만 갖게 한다.
- UI, 게임 규칙, LLM 연결, 저장 로직은 섞지 않는다.
- 예외를 숨기지 말고, 어디서 왜 실패했는지 드러나게 만든다.
- 안 쓰이는 코드는 비활성 상태로 남겨두지 않고 바로 삭제한다.
- `fallback`, 임시 우회, 죽은 분기, 더 이상 호출되지 않는 테스트 스크립트는 유지보수 비용만 늘리므로 남기지 않는다.

## 3. 파일 분리 기준

- `llmoker/game/*.rpy`
  - Ren'Py 화면, 라벨 흐름, 연출, 입력 처리만 둔다.
- `llmoker/backend/*.py`
  - 포커 규칙, 상태 전이, 저장, 기억, 로그, 정책 피드백을 둔다.
- `llmoker/backend/llm/*.py`
  - Qwen-Agent, 프롬프트, tool calling, 런타임 통신만 둔다.
- `llmoker/scripts/*.py`
  - 개발자가 수동 실행하는 CLI 테스트만 둔다.

런타임 코드가 `scripts/`에 들어가면 안 된다.

## 4. 함수 작성 기준

- 함수 하나가 여러 단계 일을 하면 중간 헬퍼로 나눈다.
- 조건문이 깊어지면 “판단”, “상태 갱신”, “로그 기록”을 분리한다.
- 같은 응답 사전이나 메시지 구조를 여러 번 만들면 헬퍼 함수로 뽑는다.
- 부수효과가 큰 함수는 이름에서 그 사실이 드러나야 한다.
  - 예: `persist_feedback`, `save_slot`, `shutdown_runtime`

## 5. 네이밍 기준

- 축약어보다 의미를 우선한다.
  - `cfg`보다 `backend_config`
  - `msg`보다 `message`
- 포커 규칙 이름은 실제 용어를 따른다.
  - `betting1`, `betting2`, `draw`, `showdown`
- LLM 관련 이름은 역할이 보이게 쓴다.
  - `build_action_prompt`
  - `build_policy_response`
  - `ensure_ready`

## 6. 독스트링 기준

모든 함수와 클래스는 사람이 읽는 설명을 남긴다.
형식은 아래를 따른다.

```python
def example(arg1, arg2):
    """
    현재 상태를 읽어서 다음 행동 후보를 정리한다.

    Args:
        arg1: 현재 상태 객체다.
        arg2: 보조 설정 값이다.

    Returns:
        다음 단계에서 사용할 행동 후보 목록이다.
    """
```

규칙:

- 첫 줄은 “무슨 일을 하는지”를 자연스러운 문장으로 적는다.
- 인자와 반환값이 실제로 있을 때만 `Args:`와 `Returns:`를 둔다.
- 인자가 없거나 반환값이 없으면 해당 헤더를 억지로 넣지 않는다.
- 대신 그런 함수는 부수효과, 호출 시점, 상태 변화까지 더 자세히 적는다.
- `함수명, 함수 동작, Args:, Returns:` 같은 자동 생성 냄새 문구는 금지한다.
- 문서용 장식이 아니라, 실제로 코드를 읽는 사람이 궁금한 내용을 적는다.
- 모듈 레벨 독스트링은 파일 역할만 짧게 설명하는 한두 문장으로 끝내도 된다.
- 함수와 클래스는 모듈 독스트링처럼 한 줄로 끝내지 말고, 필요한 맥락을 자연스러운 설명으로 풀어 쓴다.

## 7. 주석 기준

- 코드만 봐도 분명한 줄에는 주석을 달지 않는다.
- 아래 경우에만 짧은 주석을 단다.
  - 상태 전이 이유가 바로 안 보일 때
  - 외부 런타임/프로세스 경계처럼 실행 맥락이 필요할 때
  - Ren'Py 제약 때문에 우회하는 코드일 때
  - 공개 정보와 비공개 정보를 분리하는 지점일 때

좋은 주석 예:

```python
# 외부 런타임은 준비 전까지 모델 로딩 로그를 섞어 출력할 수 있다.
```

나쁜 주석 예:

```python
# 값을 변수에 넣는다.
```

## 8. Ren'Py 코드 기준

- `_`를 임시 변수명으로 쓰지 않는다.
- `renpy.say()`에 들어가는 문자열은 Ren'Py 치환 문자 충돌을 피해야 한다.
- 화면 함수는 배경 선택, 상태 문구 조합, 버튼 노출 조건을 분리한다.
- `game/*.rpy` 안에서 포커 규칙 계산을 길게 하지 않는다.

## 9. LLM 코드 기준

- 엔진은 최종 합법성 검사만 맡고, LLM은 제안만 한다.
- 프롬프트 함수, 런타임 응답 정리, 런타임 프로세스 관리는 서로 분리한다.
- 모델 원본 파일은 수정하지 않는다.
- 모델이 없거나 깨졌을 때는 공식 다운로드 경로와 배치 위치를 안내한다.

## 10. 문서 동기화 기준

코드 구조가 바뀌면 아래 문서도 같이 갱신한다.

- `README.md`
- `docs/blueprint.md`
- `docs/llm_npc_setup.md`
- `docs/qwen_agent.md`
- 필요하면 관련 세부 문서

문서 안 경로 표기는 프로젝트 루트 기준으로 적는다.

## 10.1 죽은 코드 정리 기준

- 현재 실행 경로에서 더 이상 호출되지 않는 함수는 즉시 삭제한다.
- “나중에 다시 쓸 수도 있음”은 보존 이유가 되지 않는다.
- 비활성 기능이 필요하면 문서나 이슈에 기록하고, 활성 코드에서는 제거한다.
- 테스트 스크립트도 현재 구조를 검증하지 못하면 같이 삭제하거나 갱신한다.

## 11. QA 문서 기준

- QA용 문서는 `docs/qa_YYYYMMDD.md` 형식으로 만든다.
- QA 결과 문서는 `docs/qa_reqult_YYYYMMDD.md` 형식으로 만든다.
- QA 문서는 작업 전 확인할 문제 목록과 관찰 근거를 적는다.
- QA 결과 문서는 실제로 무엇을 확인했고, 무엇이 개선됐고, 무엇이 남았는지 적는다.
- QA 문서는 기능 설계 문서와 섞지 않는다.
- 같은 날짜의 QA 문서와 QA 결과 문서는 한 쌍으로 유지한다.

## 12. 검증 경로 기준

- 자동 검증과 수동 GUI QA는 분리해서 적는다.
- 자동 검증은 창 없이 재현 가능한 문법, lint, import, 기본 런타임 확인만 맡는다.
- 실제 연출, UI 감각, 대사 품질, 지연 체감은 수동 GUI QA에서 다룬다.
- 자동 검증 진입점이 생기면 `scripts/`에 두고, 관련 문서를 함께 갱신한다.
