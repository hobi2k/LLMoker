### 3. Qwen-Agent 회고 경로에서 `NoneType` 예외가 계속 난다

상태: `부분 해결`

관찰:
- 로그에 반복적으로 아래 오류가 난다.
- `정책 피드백 생성 실패: 'NoneType' object has no attribute 'get'`

영향:
- 회고가 저장되지 않는다.
- 강화학습용 기억이 누적되지 않는다.
- 장기적으로는 행동과 대사 품질이 계속 제자리에서 맴돌게 된다.

원인 후보:
- `policy` 태스크 응답 처리 경로 어딘가에서 `None` 메시지를 여전히 `dict`처럼 읽고 있다.
- action/dialogue 경로에 넣은 방어가 policy 경로 전체에 동일하게 반영되지 않았을 가능성이 있다.
- Qwen-Agent가 반환하는 메시지 목록 중 `None`이 섞일 때, `final_assistant_text()` 이후의 JSON 추출 경로에서 다시 터질 수 있다.
- 플레이어 경험 기준으로는 현재 `policy_loop.py`의 규칙 기반 폴백이 들어가 있어 회고가 완전히 비어 버리진 않지만, 근본 원인 추적은 아직 남아 있다.

해야 할 일:
- `policy` 태스크 전용 실행 경로를 다시 추적하고, `None` 메시지 처리 누락 지점을 정확히 잡을 것
- 회고 응답 원문을 디버그 로그로 남기는 옵션을 넣어, 파서 이전 단계와 이후 단계를 분리해서 볼 것
- 회고가 실패한 경우 기억 저장을 건너뛰는 현재 동작은 유지하되, 실패 원인 자체는 더 구체적으로 남길 것

  - 사건을 찌르는 대사보다 상태를 설명하는 문장 위주

- LLM 모델은 현재 포커 게임의 규칙을 알지 못한다.
- 공개 로그나 상태 문장을 대사용 한국어로 재작성하는 단계가 부족하다.
- `사야`의 성격보다 `포커 상태 설명`이 더 강하게 입력된다.

해야 할 일:
- 대사용 입력을 `사건 요약`, `감정 방향`, `상대에게 던질 의도`로 나눌 것
- 게임 규칙에 관한 맥락을 제공할 것
- 사용자가 한 번에 알기 어려운 용어는 사용하지 않을 것
- `빠르게`, `떨어진다`, `블라인드`, `드로우 시작됐다`처럼 설명투를 유도하는 표현이 프롬프트에 남아 있는지 전수 점검할 것
- 실제 사야 대사 예시와 말투 제약을 더 정교하게 다시 설계할 것
- 포커 테이블에서 실제로 쓸 수 없는 표현과 번역투를 금지 규칙이 아니라 입력 설계 단계에서 없앨 것
- 최종 대사에 따옴표가 섞이면 실패 처리하거나 재생성하게 만들 것


### 10. 행동과 교체 이유가 지나치게 일반론적이다

상태: `미해결`

관찰:
- 행동 이유가 반복적으로 아래 패턴으로 굳는다.
  - `현재 손패는 강한 족보를 형성하지 못했고...`
  - `리스크를 최소화하기 위해 체크한다`
- 카드 교체 이유도 반복적으로:
  - `0번과 2번 카드는 족보에 기여하지 않아 교체한다`
- 19:31 로그에서도 행동 이유가 거의 모두 `현재 손패는 강한 족보를 형성하지 못했고...` 패턴으로 반복된다.
- 추가 로그에서는 사실 오류까지 섞여 있다.
  - `현재 팟이 1000원이고, 내 손패는 페어 8으로 족보가 강하므로 베팅을 진행한다.`
  - 실제 게임 로그의 팟 표기는 `칩`인데 이유 문장에는 `1000원`이 들어감
  - 현재 손패는 `7 원페어`, `Q 원페어`인데 이유는 반복해서 `페어 8`이라고 말함

영향:
- 플레이가 단조롭고, 캐릭터성이 사라진다.
- QA 관점에서도 모델이 실제 상태를 읽었는지 확인하기 어렵다.
- 모델이 현재 게임 상태를 제대로 읽지 못하고, 이전 패턴이나 잘못된 샘플을 복제하고 있다는 강한 신호다.

해야 할 일:
- 행동/교체 프롬프트에서 판별 근거를 더 구체적으로 요구할 것
- `현재 족보`, `상대 행동`, `베팅 흐름`, `드로우 직전/직후`를 실제 이유에 반영하게 만들 것
- 너무 비슷한 이유가 반복되면 디버그 로그에 경고를 남기거나 샘플을 따로 모을 것
- 출력 검증에서 현재 손패와 이유 문장 사이의 명백한 불일치도 잡을 것


### 11. 카드 교체 판단이 반복적으로 `[0, 2]`에 고정되는 경향이 강하다

상태: `미해결`

관찰:
- 제공된 로그에서 카드 교체 판단이 매우 자주 `[0, 2]`로 나온다.
- 손패 맥락이 달라도 같은 인덱스가 반복된다.
- 19:31 로그에서도 세 번 연속 `[0, 2]`가 반복됐다.
- 추가 로그에서는 더 직접적인 모순이 드러났다.
  - 시작 손패가 이미 `7 원페어`, `Q 원페어`인데도
  - 이유는 `페어가 없어 스트레이트를 위한 중간 카드가 부족합니다.` 라고 출력됨
  - 그러면서 교체 인덱스는 또 `[0, 2]`로 고정됨

가능성:
- 프롬프트가 카드 위치보다 추상적인 설명을 더 강조하고 있을 수 있다.
- 모델이 실제 카드 내용을 읽기보다 자주 본 출력 패턴으로 수렴하고 있을 수 있다.
- 현재 손패를 파싱해서 인덱스별 카드 의미를 연결하는 입력 설계가 약할 수 있다.

해야 할 일:
- 교체 판단 입력에서 각 카드 인덱스와 카드 내용을 더 명시적으로 연결할 것
- 교체 결과와 실제 손패 변화가 합리적인지 검증하는 QA 로그를 추가할 것
- 동일한 출력 패턴 반복 빈도를 계측할 것
- 이미 페어가 있을 때 `페어가 없다` 같은 모순 문장을 바로 실패 처리할 검증도 필요하다


## 구조와 로직

### 11-1. 드로우 화면에서 `교체 확정`과 `교체 없이 진행`의 차이가 불분명하다

상태: `해결`

관찰:
- 선택한 카드가 하나도 없어도 `교체 확정`으로 그대로 다음 단계로 넘어갈 수 있었다.
- 실제로는 `교체 없이 진행` 버튼과 체감 차이가 거의 없었다.

조치:
- `교체 확정`은 선택 카드가 하나 이상 있을 때만 진행되게 바꿨다.
- 선택 카드가 없으면 상태 문구와 알림으로 경고를 띄우고 현재 화면에 머문다.
- `교체 없이 진행`은 별도 경로로만 드로우 단계를 넘기게 분리했다.

검증:
- draw 화면 버튼이 `confirm_draw_selection()` / `skip_draw_selection()`으로 분기된다.
- 비GUI QA에서 compile/lint/python compile 통과


### 12. 플레이어 선택 후 봇 반응 흐름은 로그만 보면 아직도 오해하기 쉽다

상태: `부분 해결`

관찰:
- 현재는 플레이어 행동 로그가 추가됐지만, 여전히 사용자는 "중간에 행동이 연속으로 두 번 일어난다"고 느끼기 쉽다.
- 실제로는 첫 베팅, 드로우, 두 번째 베팅, 응답 흐름이 로그상 잘 드러나지 않는다.

해야 할 일:
- 각 페이즈 전환 시 `페이즈 시작`, `턴 이동`, `플레이어 대기`, `봇 대기` 로그를 더 명확히 남길 것
- `공개 로그`와 `디버그 로그`를 더 분리해 사용자가 실제 라운드 흐름을 쉽게 추적할 수 있게 만들 것


### 13. 기억/회고 파이프라인이 현재는 강화학습 기반으로 기능하지 않는다

상태: `미해결`

관찰:
- 회고가 계속 실패해서 메모리가 누적되지 않는다.
- 사용자는 "저장 안 하면 초기화되되, 저장/불러오기는 연동" 구조를 원했는데, 그 전제 위에서 가장 중요한 회고 생성이 깨져 있다.
- 19:31 로그에서도 매 라운드 다음 오류가 반복됐다.
  - `정책 피드백 생성 실패: 'NoneType' object has no attribute 'get'`

해야 할 일:
- 회고 생성 실패를 최우선 버그로 다룰 것
- 회고 성공 시 실제로 어떤 텍스트가 SQLite에 들어가는지 검증 로그를 남길 것
- 장기 기억과 단기 기억이 대사/행동에 어떻게 반영되는지 실제 샘플로 검수할 것
